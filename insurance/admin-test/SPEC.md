# 빌리지점 배서 업무 보드 — 구현 사양서 v0.1

> 이 문서는 사내 전산팀 / AI(클로드·ChatGPT 등) 개발 어시스턴트에게 풀스택 구현을 의뢰할 때 그대로 전달할 수 있도록 작성되었습니다.
> 함께 배포된 정적 목업( `villagebaby.kr/insurance/admin-test/` )이 이 문서의 시각적 레퍼런스입니다.

---

## 0. 개요

빌리지베이비 [빌리지점]은 보험 **배서(endorsement)** — 계약 변경 — 업무를 다음 흐름으로 처리한다.

1. 상담사가 Google Form을 통해 배서 요청을 접수
2. 응답이 구글 시트(엑셀DB)에 쌓이면 총무가 시트를 보고 보험사 시스템에 접속해 처리 (태아등재, 등본·동의서 첨부, 전자서명 발송 등)
3. 총무가 시트에 [완료] 마킹

### 현재 통증 포인트

| # | 통증 | 영향 |
|---|------|------|
| 1 | 단일 폼이라 업무구분별 필수정보 누락 (예: 카드변경인데 카드번호 없음) | 상담사·총무 간 추가 통신 발생, 처리 지연 |
| 2 | 바쁜 시기에 총무가 [완료] 마킹·회신을 못 줌 | 상담사가 처리 여부를 모름 → 불만 누적 |
| 3 | 상담사가 자기 제출 건의 진행 상황을 한 곳에서 볼 수 없음 | 보험사 시스템에 직접 들어가 조회하라는 안내 → 추가 불만 |

### 본 시스템의 목표

- (a) 업무구분에 따라 **필수 필드를 동적으로 강제**하여 누락을 시스템적으로 방지
- (b) 상담사가 **자기 제출 건과 처리상태를 언제든 조회** 가능 (마이페이지)
- (c) 상태가 바뀌면 **상담사에게 자동 알림** (이메일 우선, 카카오 알림톡은 추후)
- (d) 총무에게는 **처리 중심의 효율적인 보드**를 제공 (필터, 인라인 상태 변경, 메모, "내가 잡기")

### 본 시스템의 비목표 (out of scope)

- 보험사 시스템과의 직접 연동 (전자서명 발송 자동화 등) — 총무가 보험사 시스템에서 수기 처리하는 흐름은 유지
- 계약 데이터베이스와의 통합 — 본 시스템은 "배서 요청"만 다룬다
- 고객 직접 접속 — 고객은 이 시스템에 접근하지 않는다

## 1. 사용자·역할

| 역할 | 인원(예상) | 주요 액션 |
|------|----------|---------|
| **상담사** | 다수 | 배서 요청 제출, 자기 제출 건 조회, 첨부파일 보기 |
| **총무** | 소수 (2~3명) | 모든 요청 조회, 상태 변경, 메모 작성, 본인 담당 픽업 |
| **관리자** | 1~2명 | 사용자 등록·비활성화, 통계 조회 |

### 인증 (실제 구현)

- 회사 Google Workspace 계정 OAuth 로그인 권장
- 도메인 한정 (예: `@villagebaby.kr` 만 허용)
- 역할은 별도 `사용자` 시트/테이블에서 관리

### 인증 (목업)

- localStorage `currentUser` 로 모의. 헤더 우상단 드롭다운으로 역할 전환.

## 3. 업무구분 7종 — 동적 필드 사양

`업무상세_JSON` 필드에 업무구분별로 다음 키들이 들어간다.

### 3.1 태아등재 (`fetal_registration`)
```json
{
  "due_date": "2026-08-15",          // 출산예정일, ISO date, 필수
  "attached_family_register": true,   // 등본 첨부 체크, 필수
  "attached_consent": true            // 동의서 첨부 체크, 필수
}
```

### 3.2 자동이체변경 또는 카드변경 (`payment_method_change`)
```json
{
  "change_type": "card",              // "card" | "bank_transfer", 필수
  // change_type === "card":
  "card_company": "현대카드",         // 필수
  "card_last4": "1234",               // 카드번호 끝 4자리만, 필수
  // change_type === "bank_transfer":
  "bank": "국민은행",                 // 필수
  "account_holder": "김지원",         // 필수
  "account_number": "123456-78-901234", // 필수
  "reason": "기존 카드 만료"          // 변경사유, 필수
}
```

**🔒 카드번호 정책**: 전체 16자리는 어떤 경우에도 받지 않는다. 끝 4자리 + 카드사만 기록. 전체번호는 보험사 시스템에 총무가 직접 입력하는 현재 흐름을 유지 (PCI 준수 및 개인정보 최소수집 원칙).

### 3.3 담보삭제 및 추가 (`coverage_change`)
```json
{
  "add_coverages": ["암 진단비 추가", "수술비 특약"],   // 추가 담보, 배열
  "remove_coverages": ["치과 특약"],                    // 삭제 담보, 배열
  "reason": "보장 강화 요청"                            // 필수
}
```

### 3.4 이관고객관리 (`client_transfer`)
```json
{
  "transfer_reason": "담당자 퇴사",         // 필수
  "previous_advisor": "박지훈",             // 이관 전 담당자, 필수
  "new_advisor": "김지원"                   // 이관 후 담당자, 필수
}
```

### 3.5 육아 휴직 및 출산 할인 (`parental_leave_discount`)
```json
{
  "type": "parental_leave",                 // "parental_leave" | "childbirth", 필수
  "start_date": "2026-06-01",               // 필수
  "end_date": "2027-05-31",                 // 필수 (출산할인은 종료일 = 출산예정일)
  "attached_proof": true                    // 증빙서류 첨부 체크, 필수
}
```

### 3.6 계약자변경 (`policyholder_change`)
```json
{
  "previous": {
    "name": "김민준",
    "relation_to_insured": "부",
    "birth_date": "1980-03-15"
  },
  "new": {
    "name": "박서연",
    "relation_to_insured": "모",
    "birth_date": "1982-07-22"
  }
}
```

### 3.7 기타 (`other`)
```json
{
  "description": "약관대출 신청 관련 문의"   // 자유 텍스트, 필수, 최소 10자
}
```

## 4. 상태 전이

```
        ┌─────────┐
        │ (제출)  │
        └────┬────┘
             ▼
        ┌─────────┐
   ┌────│  접수   │────┐
   │    └────┬────┘    │
   │         ▼         │
   │    ┌─────────┐    │
   │    │ 진행중  │    │
   │    └────┬────┘    │
   │         ▼         │
   │    ┌─────────┐    │
   │    │  완료   │    │ (최종 상태)
   │    └─────────┘    │
   │                   │
   ▼                   ▼
┌─────────┐       ┌─────────┐
│  반려   │       │  보류   │
└────┬────┘       └────┬────┘
     │                 │
     └─── 접수로 복귀 ─┘  (상담사 추가 정보 제출 후)
```

| 전이 | 트리거 | 부수효과 |
|------|--------|---------|
| (제출) → 접수 | 상담사 제출 | 총무 그룹에 이메일 알림, 로그 기록 |
| 접수 → 진행중 | 총무 "내가 잡기" 또는 상태 변경 | 담당총무 채움, 로그 |
| 진행중 → 완료 | 총무 상태 변경 | 완료일시 채움, **상담사에게 이메일 알림**, 로그 |
| 임의 → 반려 | 총무 상태 변경 (사유 필수) | 상담사에게 반려사유 이메일, 로그 |
| 임의 → 보류 | 총무 상태 변경 (사유 필수) | 상담사에게 보류사유·필요추가정보 이메일, 로그 |
| 반려/보류 → 접수 | 상담사가 정보 보완 후 재제출 | 새 요청을 만드는 것이 아니라 기존 건의 상태 되돌림. 로그 |

## 8. 권장 구현 옵션

### 옵션 A (단기 대체용 권장) — Google Apps Script + Sheets + Drive

| 항목 | 선택 |
|------|------|
| 프론트 | `HtmlService` 템플릿 (이번 목업 HTML을 90% 재사용 가능) |
| 백엔드 | Apps Script 서버 함수 (`doGet`, `doPost`, 시트 R/W) |
| DB | Google Sheets 워크북 1개, 4개 시트 |
| 파일 저장 | Google Drive 폴더 `/배서첨부/YYYY-MM/{요청ID}/` |
| 인증 | 자동 Google Workspace SSO (도메인 한정) |
| 알림 | `MailApp.sendEmail`, 추후 카카오 알림톡은 외부 API 호출 |
| 비용 | 0원 (Workspace 라이선스 내) |
| 소요 | 약 1주 (MVP 기준) |
| 한계 | 동시 사용자 수 많아지면 성능 저하, 복잡한 UI 한계 |

**언제 옵션 A로 충분한가**: 동시 사용자 10명 내외, 일 요청 100건 이하, 통계 대시보드 단순.

### 옵션 B (정식 시스템화) — Next.js + Supabase

| 항목 | 선택 |
|------|------|
| 프론트 | Next.js (App Router) + Tailwind |
| 백엔드 | Supabase (Postgres + Auth + Storage) |
| DB | Postgres 테이블 4개 (요청·사용자·로그·알림큐) |
| 파일 저장 | Supabase Storage |
| 인증 | Supabase Auth + Google OAuth |
| 알림 | Resend / SendGrid + 카카오 비즈메시지 API |
| 호스팅 | Vercel |
| 비용 | 월 약 25~50 USD |
| 소요 | 약 3~4주 |

**언제 옵션 B가 필요한가**: 다지점 확장, 동시 사용자 50명+, 통계 분석·BI 연동.

### 마이그레이션 경로
- 옵션 A로 시작 → 데이터를 옵션 B로 export 시점에 시트 → CSV → Postgres 일괄 import
- `요청ID` 포맷이 동일하므로 외부 참조(이메일 본문 등)도 호환
